앱·서비스 이름을 짓습니다. 4개 전략별 에이전트가 후보를 생성하고, 평가 에이전트가 랭킹하며, 가용성 에이전트가 도메인·상표·다국어 의미를 점검해 한국어 추천 리포트를 출력합니다.

인자: `$ARGUMENTS` — 서비스/앱의 한 줄 컨셉 또는 브리프(파일 경로도 허용)

## 워크플로

### 1단계: 브리프 수집·확정

좋은 이름은 브리프 품질에 좌우됩니다. 다음 11개 필드로 브리프를 구성합니다:

| 필드 | 설명 |
|---|---|
| **컨셉** | 무엇을 하는 서비스/앱인가 (한 줄) |
| **핵심 편익·차별점** | 사용자가 얻는 가장 큰 가치 |
| **타깃 사용자** | 누구를 위한 것인가 |
| **톤·성격** | 진지/전문 · 친근/캐주얼 · 혁신/테크 · 프리미엄 · 귀여움 등 |
| **시장·언어권** | 한국 위주 · 글로벌(영어권) · 특정 지역 |
| **경쟁/참조 서비스** | 비슷해 보이면 안 되는 경쟁군, 참고할 톤의 서비스 (차별성·Copycat 판단 기준, 없으면 "없음") |
| **카테고리** | 속한 제품/서비스 카테고리 (상표 보호·차별성 판단 기준) |
| **향후 확장 범위** | 앞으로 넓힐 제품군/지역 (확장성 판단 기준, 없으면 "현재 범위") |
| **주 사용 표기** | 한글 / 영문 / 둘 다 |
| **선호·필수** | 포함하고 싶은 키워드, 음절 수, 어조 (없으면 "없음") |
| **반드시 피할 것** | 경쟁사명, 특정 단어, 부정 연상 (없으면 "없음") |

처리 규칙:

1. `$ARGUMENTS`가 **비어 있으면**: 사용자에게 최소 **컨셉·타깃·톤·시장/언어권·카테고리**를 물어보고 중단합니다.
2. `$ARGUMENTS`가 **파일 경로**(`.md` 등)이면 Read로 읽어 브리프를 추출합니다.
3. `$ARGUMENTS`에 컨셉이 있으면: 나머지 필드를 합리적으로 추론해 채우고 추론값에 "(추정)"을 붙입니다. 단, **이름 품질에 큰 영향을 주는 핵심 맥락 필드(시장/언어권 · 카테고리 · 톤 · 경쟁/참조 서비스)** 가 비어 있고 합리적 추론도 어려우면 그 항목만 사용자에게 묻고 중단합니다.
4. **확정 브리프 표를 제시**한 뒤, 추정값이 포함됐다면 "추정값 기준으로 바로 진행합니다 — 바꿀 점이 있으면 알려주세요"라고 **정직하게 밝히고** 진행합니다(추정이 없거나 사용자가 이미 충분히 준 경우 곧바로 진행).

표시: `## 네이밍: <컨셉 요약>` 후 확정 브리프 표.

### 2단계: 4개 생성 에이전트 병렬 실행

4개 에이전트를 **단일 메시지에 다중 Agent 호출**로 동시에 실행합니다. `subagent_type`은 `naming:nm-*` 형식(플러그인명:에이전트명)을 그대로 씁니다.

실행할 에이전트:
1. `naming:nm-descriptive` — 서술형·암시형 (의미·편익 기반)
2. `naming:nm-coined` — 조어형 (혼성어·오탈자·접사·신조어)
3. `naming:nm-metaphor` — 은유·상징·추상·어근
4. `naming:nm-experimental` — 실험적·음성상징·짧은 임팩트

아래 프롬프트의 `<브리프>`를 1단계의 **확정 브리프 11필드로 치환한 뒤** 각 에이전트에 전달합니다(플레이스홀더를 리터럴로 보내지 마세요). 4개 모두 같은 브리프를 받습니다.

```
다음 브리프로 앱·서비스 이름 후보를 생성해주세요. 4개 에이전트가 각기 다른 전략으로 병렬 생성 중이므로 본인 레인의 후보만 만드세요. 브리프의 경쟁/참조 서비스와 시각적·음성적으로 닮지 않게 차별화하고, 주 사용 표기(한글/영문)에 맞는 후보를 우선하세요. 후보 8~12개 + 추천 Top 2를 반환하세요.

<브리프>
- 컨셉:
- 핵심 편익·차별점:
- 타깃 사용자:
- 톤·성격:
- 시장·언어권:
- 경쟁/참조 서비스:
- 카테고리:
- 향후 확장 범위:
- 주 사용 표기:
- 선호·필수:
- 반드시 피할 것:
```

### 3단계: 후보 풀 취합·중복 제거

각 생성 에이전트는 `| 이름 | 레인 | 기법 | 발음 | 근거 | 추천 |` 표를 반환합니다. 이 표들에서 후보를 추출해 모두 모읍니다. 대소문자 무시 및 거의 동일한 변형은 하나로 합치되, **각 후보의 레인 태그(서술형·암시형 / 조어형 / 은유·상징 / 실험적)와 발음·근거를 보존**합니다. 중복 제거는 이 단계에서 책임지며, 평가 에이전트에는 정리된 풀을 넘깁니다. 중복 제거 후 후보 수를 한 줄로 표시합니다(예: "총 32개 후보 수집 → 중복 제거 후 28개").

### 4단계: 평가 에이전트 실행

`naming:nm-evaluator`를 1회 실행합니다. 입력으로 **확정 브리프 + 중복 제거된 후보 풀 전체**(레인 태그 포함)를 전달합니다. 평가 에이전트는 Neumeier 7기준 + SMILE/SCRATCH로 채점·랭킹하고 **숏리스트(상위 8개 내외) + 최종 추천 Top 3~5**를 반환합니다.

### 5단계: 가용성 에이전트 실행 (숏리스트만)

`naming:nm-availability`를 1회 실행합니다. 입력으로 **4단계 숏리스트 이름들 + 브리프의 카테고리 · 경쟁/참조 서비스 · 시장·언어권(상표권역) · 주 사용 표기**를 전달합니다(상표 충돌은 카테고리 없이 판단 불가하므로 반드시 함께 넘깁니다). 가용성 에이전트는 이름마다 도메인·상표(카테고리 기반 Nice 류)·앱스토어/핸들·다국어 의미를 공식 출처 우선으로 점검하고 신호등(🟢/🟡/🔴) 플래그를 반환합니다.

(숏리스트가 비었거나 후보가 너무 적으면 이 단계를 건너뛰고 그 사실을 리포트에 명시합니다.)

### 6단계: 최종 리포트 출력 (한국어)

평가·가용성 결과를 합쳐 다음 형식으로 한국어로 출력합니다.

```
## 네이밍 리포트: <컨셉 요약>

### 확정 브리프
(11필드 표)

### 점수 Top (평가 기준 · 가용성 미반영)
평가 숏리스트 상위 3~5개를 나열: 이름 — 총점 · 한 줄 강점.

### 🏆 최종 추천 (가용성 반영)
점수 Top을 **가용성 신호등으로 재정렬**합니다 — 🔴(강한 충돌)는 추천에서 제외하거나 맨 아래로 강등, 🟡는 주의 표기, 🟢 우선. 점수는 높지만 가용성이 위험한 이름이 자동 1위가 되지 않게 합니다.
1. **<이름>** — 총점 XX · 🟢/🟡/🔴
   - 컨셉 적합 이유: ...
   - 발음: ...
   - 가용성: 한 줄 요약 (위험 시 사유·권장 조치)
(2, 3 ...)

점수 1위와 추천 1위가 다르면 그 이유(가용성)를 한 줄로 밝힙니다.

### 전체 랭킹
(평가 에이전트의 랭킹 표를 그대로 — 순위 · 이름 · 레인 · 7기준 점수 · 보정 · 총점. 후보별 강점/약점은 위 추천 섹션과 평가 숏리스트에 담기므로 표에는 넣지 않습니다.)

### 전략별 후보 풀 (부록)
- 서술형·암시형: ...
- 조어형: ...
- 은유·상징: ...
- 실험적: ...

### 다음 단계 체크리스트
- [ ] 도메인 정식 조회 (WHOIS/레지스트라) 및 .com 확보
- [ ] 상표 정식 조회 (KIPRIS / USPTO / WIPO) — 해당 Nice 분류
- [ ] 소셜 핸들·앱스토어 중복 확인
- [ ] 주요 언어권 현지화 검수
- [ ] 타깃 사용자 발음·연상 테스트

> ⚠️ 가용성 점검은 웹 검색 기반 **참고 단서**이며 법적 보증이 아닙니다. 최종 확정 전 정식 조회가 필요합니다.
```

## 규칙

- 모든 사용자 출력은 한국어로 합니다.
- 생성 에이전트는 본인 레인만 생성하며, 실재 유명 브랜드명·경쟁사명을 베끼지 않습니다.
- 평가·가용성은 분리된 단계입니다. 평가는 이름 자체 속성을, 가용성은 실제 충돌 단서를 봅니다.
- 단계 순서를 지킵니다: 생성(병렬) → 취합·중복제거 → 평가 → 가용성(숏리스트) → 리포트. 평가는 후보 풀 전체가 모인 뒤 실행합니다.
- 점수 Top은 평가 숏리스트(=가용성 점검 대상) 안에서 뽑고, **최종 추천은 점수 Top을 가용성 신호등으로 재정렬**한 결과입니다. 🔴 후보는 추천 상위에 올리지 않되 점수 Top에는 남겨 투명하게 보여줍니다.
- 가용성 결과는 항상 "참고용·법적 보증 아님"을 명시합니다.
