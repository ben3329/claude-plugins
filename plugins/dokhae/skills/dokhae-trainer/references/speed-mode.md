# speed 모드 — 속독 + 이해도

읽기 **속도**와 **이해도**를 동시에 측정해, "빠르면서도 정확한" 읽기 균형을 훈련한다.
공유 규칙은 [`_spec.md`](_spec.md), 지문 생성 절차는 [`passage-generation.md`](passage-generation.md)를 따른다.

## 1. 사용 시점·목표

- 사용 시점: 내용은 이해하는데 **읽는 속도가 느릴 때**, 또는 빠르게 읽지만 **놓치는 게 많을 때**.
- 목표: 시간을 측정한 1회 통독으로 지문을 읽고, **재독 없이** 이해도 문제를 풀어
  자신의 분당 글자수(cpm)와 이해도를 함께 확인한다.
- 핵심 메시지: **빠른데 틀리면 의미가 없고, 정확한데 너무 느려도 곤란하다.** 둘의 균형이 점수다.

## 2. 시간 측정 핸드셰이크 (가장 중요)

측정의 신뢰성이 모드 전체를 좌우한다. 아래 순서를 **반드시** 지킨다.

1. **지문을 미리 완성**해 둔다(`passage-generation.md` 기준, 기본 level 3). 아직 화면에 보이지 않게 한다.
   - 측정 노이즈가 분량 대비 작아지도록 **지문은 너무 짧지 않게**(권장 `char_count` ≥ 800, 즉 **L3 이상**) 둔다.
2. 학습자에게 안내만 먼저 제시한다(지문·문제 노출 금지):
   > "준비되면 **'시작'** 이라고 입력하세요. 그 즉시 지문이 나타나고 타이머가 시작됩니다.
   > 끝까지 한 번에 읽고, 다 읽는 **순간 바로 '완료'** 라고 입력하세요. **재독은 없습니다.**"
3. 학습자가 **'시작'** 을 입력하면, **같은 응답 안에서**:
   - 먼저 Bash로 시작 시각을 찍는다: `date +%s`  → 이 값을 **t0**으로 기억(초 단위).
   - 같은 응답에서 곧바로 **지문만** 노출한다(문제·정답 절대 금지, `_spec.md` 6절).
4. 학습자가 **'완료'** 를 입력하면, **그 응답에서 다른 어떤 작업보다 먼저** Bash로 종료 시각을 찍는다:
   `date +%s` → **t1**. (설명·계산·이해도 문제는 t1을 찍은 *다음에* 한다.)

> ⚠ **측정 편향(반드시 학습자에게 알릴 것)**: 이 방식은 정밀 스톱워치가 아니다.
> - t0→지문 노출 사이에는 모델이 지문을 출력하는 시간이,
> - **'완료' 입력→t1 사이에는 입력 왕복(network)+모델 반응 지연이 매번 더해진다.**
>
> 이 지연들은 항상 `elapsed_sec`를 **늘리는 한 방향**이라 cpm을 **체계적으로 낮춘다**. 그러므로 cpm
> **절대값보다 밴드(느림/보통/빠름)와 본인 추이**로 해석한다(4·6절). 짧은 지문일수록 지연 비중이 커지므로
> (2-1의 최소 분량 권장 이유), 측정을 더 신뢰하려면 분량을 키운다.
> 또한 t0를 찍기 전에 지문이 노출되면 측정이 무효다. **반드시 `date +%s`(t0) → 지문** 순서.

## 3. 계산

```
elapsed_sec = t1 - t0                       # t0·t1은 초 단위(date +%s)
cpm         = char_count / (elapsed_sec / 60)
```

- 계산은 초 단위로 하고, 기록(`elapsed_sec`)에는 정수 초로 적는다. cpm도 정수 반올림.
- `char_count`는 **공백 제외 한글 기준 글자수**(`_spec.md` 2·7절)로 지문 생성 시 이미 확정돼 있다.
- `elapsed_sec`가 비정상적으로 작으면(예: 글자수 대비 cpm이 밴드 상한을 크게 초과) 오입력 가능성을
  알리고, 1회 재측정을 제안한다(통독을 안 하고 '완료'를 누른 경우 등).

## 4. 한국어 묵독 속도 참고 밴드 (참고치)

아래는 **절대 기준이 아니라 참고치**다. 지문 난이도·생소함에 따라 자연히 달라진다.

| 밴드 | cpm(공백 제외) | 해석 |
|------|----------------|------|
| 느림 | < 500 | 또박또박 읽음. 속도 훈련 여지 큼 |
| 보통 | 500–800 | 일반적인 묵독 속도 |
| 빠름 | 800–1200 | 숙련된 독자 수준 |
| 매우 빠름 | > 1200 | 매우 빠름. **이해도 동반 여부를 반드시 확인** |

- 속도만으로 잘 읽었다고 보지 않는다. 같은 cpm이라도 **이해도(comp_accuracy)** 가 낮으면 무의미하다.
- 위 수치는 난이도·생소함·개인차로 쉽게 **±20% 변동**하며, 2절의 측정 편향까지 더해지므로 **등급 라벨이
  아니라 대략의 위치**로만 본다. 한 칸 경계 부근의 차이(예: 480 vs 520)에는 의미를 두지 말고,
  **직전 본인 기록과의 변화(6절 추이)** 를 우선 본다.

## 5. 이해도 검증 (읽기 측정이 끝난 뒤에만)

- 이해도 문제는 **'완료' 입력 후 t1을 찍은 다음에만** 제시한다(`_spec.md` 6절). 측정 전 노출 금지.
- 문제는 **3~4개**, **핵심 위주**로 낸다(중심 화제·핵심 주장·문단 관계·중요 함의). 지엽적 암기는 피한다.
- **재독 금지**: 지문을 다시 보여주지 않고, 기억에 의존해 답하게 한다.
- 채점은 **지문 내부 근거**로 한다. 각 문항에 정답·간단한 근거 문단(`[1] [2]`)을 표로 제시한다.
- 이해도 계산: `comp_accuracy = comp_correct / comp_q`.

| 문항 | 학습자 답 | 정답 | 근거 | 판정 |
|------|-----------|------|------|------|
| 1 | … | … | `[2]` … | O/X |

## 6. 종합 판정 (속도 × 이해도 균형)

속도와 이해도를 **함께** 보고 판정한다. 한쪽만 보지 않는다.

| 상황 | 판정 | weak_tags |
|------|------|-----------|
| 빠름 + 이해도 낮음(예: comp_accuracy < 0.6) | "급하게 읽어 정확도를 잃음" | `정확도부족` (놓친 유형 따라 `핵심누락`/`사실적이해` 추가) |
| 느림(밴드 '느림') + 이해도 보통 이상 | "정확하나 속도 보완 필요" | `속도부족` |
| 빠름·보통 + 이해도 높음 | "균형 양호" | (없거나 1개) |
| 느림 + 이해도도 낮음 | "속도·정확도 모두 보완" | `속도부족` + `정확도부족` |

- 이해도 오답 유형에 맞춰 표준 태그(`사실적이해`/`추론`/`구조파악`/`핵심누락` 등, `_spec.md` 5절)를 함께 단다.
- **추이 코멘트**: `records.jsonl`의 직전 `speed` 세션(있으면 2~3개)과 비교해
  cpm·comp_accuracy가 올랐는지/내렸는지 한 줄로 짚는다. 예: "지난 720cpm·0.75 → 이번 860cpm·0.50:
  속도는 올랐으나 이해도 급락. 다음엔 속도를 약간 늦춰 정확도부터 회복하세요."

## 7. 기록 필드 (speed)

`_spec.md` 8절 절차대로 `records.jsonl`에 JSON 한 줄로 append한다. speed에서 채우는 필드:

```json
{"date":"2026-06-30","time":"15:42","mode":"speed","area":"과학","topic":"광합성의 명반응","level":3,
 "char_count":920,"q_count":null,"correct":null,"accuracy":null,"scores":null,
 "elapsed_sec":77,"cpm":717,"comp_q":4,"comp_correct":2,"comp_accuracy":0.5,
 "weak_tags":["정확도부족","핵심누락"],"note":"보통 속도지만 이해도 0.5. 핵심 함의 2문항 놓침"}
```

- quiz·summary 전용 필드(`q_count`/`correct`/`accuracy`/`scores`)는 **`null`**.
- `cpm`은 정수 반올림 권장. `comp_accuracy`는 0–1.

## 8. 짧은 예시 (흐름 + 계산 + 판정)

```
[준비 안내 — 지문 미노출]
Claude: 준비되면 '시작'을 입력하세요. 지문이 나오고 타이머가 시작됩니다.
        다 읽는 순간 바로 '완료'를 입력하세요. 재독은 없습니다.
학습자: 시작
Claude: (Bash: date +%s → t0=1750000000)  ← 같은 응답에서 t0 먼저
        [지문] (광합성 명반응, char_count=920, level 3)   ← 지문만 노출
학습자: 완료
Claude: (Bash: date +%s → t1=1750000077)  ← 다른 작업보다 먼저 t1

[계산]  elapsed_sec = 1750000077 - 1750000000 = 77초
        cpm = 920 / (77/60) ≈ 717  → 밴드 '보통'. (측정은 근사값입니다.)

[이해도 — 측정 종료 후 제시, 재독 없음] 4문항 → 2정답
        comp_accuracy = 2/4 = 0.5

[종합 판정] 속도는 보통(717cpm)이나 이해도 0.5로 낮음 → '정확도부족'.
            오답은 핵심 함의 2문항(핵심누락). 지난 0.75 → 0.50으로 이해도 하락.
            다음엔 속도를 조금 늦춰 핵심 문장에 표시하며 읽기를 권장.

[기록] records.jsonl에 1줄 append → "기록됨". 다음 포커스: 정확도부족 / 핵심누락.
```
